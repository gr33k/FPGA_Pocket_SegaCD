# Task 6S: Docker Quartus Build-Host Prep

## Why Docker is used

Quartus is not available on the local host for this repository, and the goal is to
keep build-host execution deterministic. This is why this milestone adds a Docker
host scaffold so a Fedora NAS (or any Quartus-capable machine) can run the same
APF scaffold checks using one reproducible command surface.

## What was added

- `docker/` workspace helper with:
  - `docker/quartus/Dockerfile.ubuntu-quartus-base`
  - `docker/quartus/README.md`
  - `docker/quartus/.gitignore`
- Repo-root helper docs:
  - `docs/FEDORA_NAS_DOCKER_QUARTUS_WORKFLOW.md`
  - `docs/QUARTUS_DOCKER_INSTALLER_POLICY.md`
  - `docs/DOCKER_QUARTUS_VALIDATION_STATUS.md` (generated by checker script)
- Tooling scripts:
  - `tools/check_quartus_docker_workflow.sh`
  - `tools/run_quartus_docker_dryrun.sh`
- `.gitignore` entries for installer artifacts and Quartus outputs.
- Existing Quartus handoff/project-flow docs updated to include Docker preflight path.

## Why Quartus is not committed

Intel Quartus is proprietary tooling. The workflow intentionally never commits:

- Quartus installer payloads
- Installed Quartus directories
- Quartus license files
- Quartus-generated bitstreams or intermediate analysis outputs

This keeps the repo legal and clean while still making build-host execution feasible.

## How this unblocks the next step

- A host with Quartus can now run a known base image and validate discovery
  (`quartus_map` presence) before attempting any scaffold analysis/elaboration commands.
- The same repository can be checked on/off-host without local Quartus assumptions.

## Remaining blockers after this task

- No real Quartus installation exists in this repository host path yet.
- No generated core analysis output has been produced.
- No real runtime compile/fit has been run in this milestone.
